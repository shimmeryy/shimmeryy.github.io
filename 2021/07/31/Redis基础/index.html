Redis基础

1. NoSQL

NoSQL（Not only SQL），不仅仅只是SQL，是一种非关系型数据库。随着互联网体量的逐渐增大，传统的关系型数据库在应对大规模数据高并发的应用场景显得力不从心，比如热门视频排行、订单超时等应用场景在传统的关系型数据库中实现较为复杂。众所周知，碰到问题了就可以考虑“加一层”，NoSQL的出现解决了这类问题。

NoSQL主要有四类：

第一种是key-value型存储数据库，主要用Hash表维护，代表是Redis。

第二种是文档型数据库，以文档形式存储，代表是MongoDB。

第三种是列存储数据库，一般用来应对分布式存储的海量数据，代表是HBase。

第四种是图形数据库，使用灵活的图形模型存放关系。代表是Neo4j。

分类 Examples举例 典型应用场景 数据模型 优点 缺点
键值（key-value） Tokyo Cabinet/Tyrant， Redis， Voldemort， Oracle BDB 内容缓存，主要用于处理大量数据的高访问负载，也用于一些日志系统等等。 Key 指向 Value 的键值对，通常用hash table来实现 查找速度快 数据无结构化，通常只被当作字符串或者二进制数据
列存储数据库 Cassandra， HBase， Riak 分布式的文件系统 以列簇式存储，将同一列数据存在一起 查找速度快，可扩展性强，更容易进行分布式扩展 功能相对局限
文档型数据库 CouchDB， MongoDB Web应用（与Key-Value类似，Value是结构化的，不同的是数据库能够了解Value的内容） Key-Value对应的键值对，Value为结构化数据 数据结构要求不严格，表结构可变，不需要像关系型数据库一样需要预先定义表结构 查询性能不高，而且缺乏统一的查询语法。
图形(Graph)数据库 Neo4J， InfoGrid， Infinite Graph 社交网络，推荐系统等。专注于构建关系图谱 图结构 利用图结构相关算法。比如最短路径寻址，N度关系查找等 很多时候需要对整个图做计算才能得出需要的信息，而且这种结构不太好做分布式的集群方案。

2. Redis

Redis（Remote Dictionary Server）是一个高性能的Key-Value内存型数据库，支持五大基本数据类型和三大特殊数据类型，同时支持持久化（RDB，AOF），它可用作数据库、缓存、消息中间件，最主要用来作缓存。

Redis在6.0版本之前，主要体现为单线程。Redis基于Reactor模式开发了网络事件处理器即文件事件处理器，由多个socket、IO多路复用程序、文件事件分派器、事件处理器所组成。由于文件事件分派器的队列的消费是单线程的，所以Redis称为单线程模型。

一般来说Redis的瓶颈不在于CPU，而是内存和网络。同时，使用单线程后可维护性高，如果使用多线程，虽然性能表现较好，但是由于存在上下文切换，带来了一系列并发读写的问题，同时加锁也可能会导致性能消耗。所以之前Redis一直使用单线程模型。

在Redis6.0发布后，可以在conf文件中配置开启多线程模式。（但默认仍是不开启）我们前面说过，Redis的瓶颈在于内存和网络，内存我们可以通过money加内存来解决，但是网络的性能优化比较棘手。网络IO读写在Redis执行过程占据了大量的时间，如果我们在网络IO读写中使用多线程的处理方式，能够充分的提高网络IO性能，从而提升整个Redis的性能。

3. Redis基本命令

redis-server configpath/config# 连接redis服务端

redis-cli -p 6379 						# 选择以6379为端口号的客户端

select 1 											# 选择数据库1

flushdb												# 清空当前数据库

flushall											# 清空所有数据库

keys * 												# 获取所有key

dbsize												# 查看当前数据库的大小

exist keyname									# 根据键名判断是否存在键值对，是返回1，否返回0

set key value									# 设置键值对

get key												# 根据key获取key中的value

move key 2										# 将key键值对移到数据库2

expire key 10									#	设置key键值对过期时间为10s

type key											# 查看key键所对应值的类型

4. 基本数据类型

4.1 String

set key1 zhangsan

append key1 azhe # 在key1后追加字符串azhe
append key azhe  # 如果key不存在，则创建一个key为azhe

set age 1

incr age 			# 将age的值自增1
incrby age 10 # 将age的值自增10
decr age 			# 将age的值自减1
decrby age 10 # 将age的值自减10

getrange key1 0 1  # 获取第0～1个字符
getrange key1 0 -1 # 获取值的全部内容 在其他数据类型中常用

setrange key1 1 hangergou # 从第1个字符（从0开始）替换，key1变成了zhangergou

setex key3 30(ttl) "hello" # 设置key3 过期时间为30s 值为hello

setnx key3 "hello" # 设置key3为hello 如果不存在才创建，经常用于分布式锁

mset k1 v1 k2 v2 k3 v3 # 设置多个键值对
mget k1 k2 k3					 # 获取多个值

msetnx k1 v1 k2 v2 # 如果k1，k2都不存在该语句才能执行成功

getset key "azhe"  # 先获取key的值返回，再设置key的值为azhe

mset user1:name zhangsan user1:age 10 # key设置为userx:{prop}来存储对象

String类型的主要应用场景：可以做浏览量计数器、存储多单位的数量、存储对象的缓存等。

4.2 List

Redis的List底层是linkedlist、ziplist、quicklist，总之是个双向链表。即List可以实现队列、栈、阻塞队列的功能。list的命令以l（left）和r（right）开头，核心是push和pop。

lpush mylist zhangsan

lpush mylist wangwu

lrange mylist 0 -1  # 获取mylist里的所有值

127.0.0.1:6379> lrange mylist 0 -1
1) "wangwu"
2) "zhangsan"

# 我们发现lpush是头插法，可以把链表横过来看，相当于l（left）往左边插数据

lpop mylist # pop左边的元素即“wangwu”

lindex mylist 0 # 获取下标为0的值 没有rindex哦

llen mylist # 获取list的长度

lrem mylist 1 zhangsan # 从头部开始移除1个value为zhangsan的

ltrim mylist 0 1 # 截取指定下标范围

linsert mylist before/after "value1" "value2" # 将value2插入到value1的前或者后面

rpoplpush mylist otherlist # 移除mylist的最后一个元素push到otherlist的最前面

lset mylist 1 wangwu # 根据index设置value

List类型的主要应用场景为排行榜，朋友圈点赞最新消息等。

4.3 Set

Set是一种无序不可重复集合，底层是intset和hashtable实现。指令开头为s。

sadd myset hello
sadd myset redis

smembers myset  # 查看myset里所有的值

sismember myset hello # 查看hello是否在集合中

scard myset # 查看myset集合中元素的个数

srem myset hello # 移除hello元素

srandmember myset 2 # 随机抽取myset中的两个元素

spop myset 2 # 随机移除myset中的两个元素

sdiff key1 key2  # 获取key1和key2的差集
sinter key1 key2 # 获取key1和key2的交集
sunion key1 key2 # 获取key1和key2的并集

Set的应用场景主要有共同好友、关注、粉丝、感兴趣的人的集合，音乐列表的随机展示等。

4.4 Hash

Hash是key-map集合，底层是ziplist和dict。命令以h开头。

hset myhash name zhangsan password 123456 # 设置key为myhash，map为两个key-value键值对

hget myhash name # 获取myhash的map中name这个key所对应的value

hmset myhash # 批量设置键值对 好像跟hset一样呀.. 查了查应该就是一样的

hmget myhash name password # 批量获取

hgetall myhash # 获取所有键值对

hdel myhash name # 删除指定键值对

hexists myhash name # 判断是否存在指定键

hkeys myhash # 获取所有key

hincrby myhash age 10 # 自增某一字段

hsetnx myhash name zhangsan # 不存在则创建，否则失败

Hash的应用场景主要有购物车的实现，对象中某些频繁变化的属性的存储。

4.5 Zset

Zset是一种有序集合，底层是skipList，增加了一个值score来记数。

zadd salary 100 zhangsan 200 wangwu # 创建salary集合，先记录score再记录value。

zrange salary 0 -1 # 获取salary里所有的value

zrangebyscore salary 0 200 (min max) # 在0到200范围内排序获取value 升序

zrevrange salary 200 0 (max min) # 降序

zrem salary zhangsan # 移除zhangsan

zcard salary # 获取集合中元素的个数

zcount salary 0 100 # 获取指定区间内集合的个数

Zset的应用场景主要有排行榜实现。

5. 特殊数据类型

5.1 geospatial

geospatial类型用于存放地理位置的经纬度坐标，同时可以获取两个对象之间的距离，指定半径范围内的所有对象，geospatial的底层是Zset，可以使用Zset中的所有指令。

geoadd china:city 116.4 39.9 beijing # 添加beijing的地理位置

geopos china:city beijing # 获取指定位置经纬度

getdist china:city beijing shanghai # 计算两个位置之间的直线距离

getradius china:city 110 40 100 km (withcoord) # 以110，30经纬度为圆心，100km为半径，获取所有对象，加withcoord则同时显示筛选对象的经纬度，加withdist则同时显示直线距离

getradiusbymember china:city beijing 100km # 以为beijing为圆心，100km为半径，获取所有对象

gethash china:city beijing # 获取其hash值

5.2 Hyperloglogs

Hyperloglogs是用于统计基数的数据结构。基数为多个数据集中不重复的元素，一般用于统计网站访问人数（一个人多次访问只算一次）。Hyperloglogs有一定的错误率，使用时候必须要能够有一定的误差容忍度。

pfadd mykey1 a b c d

pfadd mykey2 b e f

pfmerge mykey mykey1 mykey2 # 将mykey1和mykey2的不重复并集merge到mykey上

pfcount mykey # 获取数量为6

5.3 Bitmaps

Bitmaps为位图，只能存储0和1，操作二进制来进行记录，很节省字节空间。可以应用在一些只有两个状态的场景，比如一年打卡365天，打卡只有1（成功）和0（失败）两个状态，只需要365个0和1记录即可。

setbit sign 0 1 # 下标为0的数据为1

getbit sign 0 # 获取下标为0的数据

bitcount sign # 获取sign中为1的数据的总数

6. Redis事务

6.1 事务

Redis事务与其他事务有所区别。Redis只有单条命令保证原子性，事务本身不保证原子性。（虽然关于原子性有点争议…）

Redis事务的本质是一组命令的集合。这个集合在执行时会在一个队列里一次执行完毕，所有的命令都被序列化，按顺序执行。

127.0.0.1:6379> multi # 事务开启
OK
127.0.0.1:6379(TX)> set k1 v1
QUEUED
127.0.0.1:6379(TX)> set k2 v2
QUEUED
127.0.0.1:6379(TX)> exec # 执行事务
1) OK # 执行结果
2) OK

discard # 在exec前使用discard可以取消事务

6.2 异常处理

Redis事务对于异常的处理有两种机制：

第一种是Redis指令使用错误，这将导致整个事务都不会执行。

127.0.0.1:6379> multi
OK
127.0.0.1:6379(TX)> set k1 v1
QUEUED
127.0.0.1:6379(TX)> sett k2 v2
(error) ERR unknown command `sett`, with args beginning with: `k2`, `v2`,
127.0.0.1:6379(TX)> exec
(error) EXECABORT Transaction discarded because of previous errors.
127.0.0.1:6379> get k1
(nil)

第二种是RunTimeException即运行时异常，这时候只有出错的语句不被执行，其他的语句正常执行。

127.0.0.1:6379> multi
OK
127.0.0.1:6379(TX)> set k1 v1
QUEUED
127.0.0.1:6379(TX)> set k2 v2
QUEUED
127.0.0.1:6379(TX)> incr k1
QUEUED
127.0.0.1:6379(TX)> exec
1) OK
2) OK
3) (error) ERR value is not an integer or out of range
127.0.0.1:6379> get k1
"v1"

6.3 乐观锁

悲观锁：很悲观，认为什么时候都会出问题，无论做什么都要加锁。

乐观锁：很乐观，认为什么时候都不会有问题，不会上锁。但是在更新数据的时候会根据version判断一下期间是否有人更改这个数据，比较与上次version是否一致。

在Redis中，我们可以使用watch指令监视对象，使用乐观锁机制。

同时我们可以使用unwatch来解除监视。如果调用了exec和discard指令，那么会自动unwatch。

6.4 不支持回滚

首先，MySQL 和 Redis 的定位不一样，一个是关系型数据库，一个是 NoSQL。

MySQL 的 SQL 查询是可以相当复杂的，而且 MySQL 没有事务队列这种说法，SQL 真正开始执行才会进行分析和检查，MySQL 不可能提前知道下一条 SQL 是否正确。所以支持事务回滚是非常有必要的~

但是，Redis 使用了事务队列来预先将执行命令存储起来，并且会对其进行格式检查的，提前就知道命令是否可执行了。所以如果只要有一个命令是错误的，那么这个事务是不能执行的。

Redis 作者认为基本只会出现在开发环境的编程错误其实在生产环境基本是不可能出现的(例如对 String 类型的数据库键执行 LPUSH 操作)，所以他觉得没必要为了这事务回滚机制而改变 Redis 追求简单高效的设计主旨。

7. Jedis

我们可以使用Java来操作Redis，Jedis是Redis官方推荐的Java连接开发工具，使用时候只需要导入对应的依赖即可。

<!-- https://mvnrepository.com/artifact/redis.clients/jedis -->
<dependency>
    <groupId>redis.clients</groupId>
    <artifactId>jedis</artifactId>
    <version>3.6.3</version>
</dependency>

连接数据库：

public static void main(String[] args) {
  Jedis jedis = new Jedis("127.0.0.1",6379);
  System.out.println(jedis.ping());
}

Jedis封装好了所有的Redis指令，使用时候创建对象后直接干就完事。
